## Short Project Description

This paper evaluates next-action selection under defective user inputs. Given a user query and optional evidence, a model must first choose whether to answer, ask for clarification, challenge a false or stale premise, or abstain when support is insufficient. The work is an evaluation of action calibration under a documented ontology, not a deployment policy and not a claim that assistants should be more confrontational in general.

## Data Sources And Provenance

Current examples are converted from benchmark-style or curated public-source material, not from private user logs or personally collected conversations. The active Day-1 evidence bundle covers answerable controls, false-premise prompts, stale-premise prompts, and conflicting-evidence questions. Stale-premise examples use public, checkable facts and are framed as premise-defect cases where the helpful first action is to correct the premise before answering.

## Risks, Misuse, And Limitations

Potential risks:

- Over-answering can reinforce false or stale assumptions.
- Over-challenging or over-abstaining can make assistants obstructive on ordinary answerable queries.
- Utility could be misread as a universal user-cost model if reported without disaggregated metrics.
- Ambiguous-intent and insufficient-evidence cases are represented in the paper-facing split, but `ask` and hard `abstain` claims should remain slice-scoped because source diversity and model-analysis depth are thinner than answer/challenge boundaries.
- Reasoning-model results are scoped to the completed local checkpoint matrix and prompt/parsing protocol.

Mitigations:

- Report action accuracy, over-answer rate, answer-supported accuracy, per-slice metrics, JSON parse rate, and utility together.
- State that utility encodes an asymmetric harm ordering, not a universal user-cost model.
- Keep `ask` and hard `abstain` claims scoped to the documented ambiguous-intent and insufficient-evidence slices; the deepest active evidence remains answer-vs-challenge and evidence-conflict calibration.
- Present `decision_first` as an exploratory calibration lever, not a deployment-ready controller.
